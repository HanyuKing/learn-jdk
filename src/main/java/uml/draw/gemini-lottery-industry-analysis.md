# gemini-lottery-industry-analysis.md

## 1. 核心共识：放弃强一致性，追求最终一致性

在阿里巴巴、腾讯、美团、字节跳动等大厂的后端架构中，针对抽奖、秒杀等高并发场景，**绝对的强一致性（如 2PC、XA）由于性能低下几乎被弃用**。

大厂的共识是：通过**最终一致性（Eventual Consistency）**方案，在保证极高性能的同时，通过可靠的消息流转和补偿机制，确保数据在短时间内（通常是毫秒级）达到一致。

---

## 2. 针对“高并发写”（抽奖/库存预占）的业界方案

这是最贴合当前 `executeDraw` 场景的方案，核心目标是：**Redis 挡在前面抗流量，DB 在后面保落账。**

### 方案 A：Redis 预扣 + MQ 异步落盘（阿里/京东标准模型）
这是处理双 11 秒杀、抢红包最常用的模型。
- **流程**：
    1. **前端层**：Redis Lua 脚本原子执行库存校验、扣减。
    2. **中间层**：Redis 成功后，立即向 **可靠消息队列（如 RocketMQ/Kafka）** 发送一条扣减成功的消息。
    3. **后端层**：DB 消费者异步拉取消息，执行 DB 事务（扣余额、写抽奖记录）。
- **一致性保障**：
    - 依靠 MQ 的 **“至少投递一次（At Least Once）”** 机制。如果 DB 写入失败，消费者会不断重试。
    - 如果重试多次仍失败（如余额被冻结），则发送**反向补偿消息**回填 Redis 库存。

### 方案 B：事务消息（RocketMQ 专属模型）
常用于美团、字节内部对资产扣减要求极高的场景。
- **流程**：
    1. 应用发送 **Half Message（半事务消息）** 到 MQ。
    2. 执行本地 DB 事务（扣余额、写记录）。
    3. 根据本地事务结果告知 MQ 是 **Commit** 还是 **Rollback**。
    4. 若 Commit，消息才会投递给下游，下游执行 Redis 状态同步或履约发放。
- **一致性保障**：MQ Server 会定时回查应用：“刚才那个半事务消息对应的本地事务到底是成功还是失败？”，从而解决断网等极端情况下的不一致。

### 方案 C：本地消息表（分布式事务经典方案）
在不引入复杂 MQ 时的首选方案，利用数据库事务保证“业务操作”与“消息发送”的原子性。
- **流程**：
    1. 在同一个 DB 事务中，插入业务记录（抽奖流水）并插入一条 **`event_log`（本地消息表）**。
    2. 事务提交后，异步线程轮询 `event_log` 表，将事件推送到 Redis 或触发履约逻辑。
- **一致性保障**：利用数据库的原子性，确保“消息”一定会被持久化。只要消息在，后台 Worker 就能通过不断重试实现最终一致。

## 3. 针对“高并发读”（配置/奖品信息）的业界方案

针对运营修改了后台配置，前端缓存需要实时更新的场景。

### 方案 D：基于 Binlog 的异步订阅（阿里 Canal / 字节 Databus）
大厂目前的主流做法，将业务代码与缓存刷新彻底解耦。
- **流程**：
    1. 业务系统正常更新 MySQL。
    2. **Canal / Databus** 伪装成 MySQL Slave，实时读取 **Binlog** 日志。
    3. 解析变更后，推送到 MQ，由专门的 **Cache Service** 消费并刷新 Redis。
- **优势**：业务代码不需要关注缓存一致性，即便主业务逻辑崩溃，缓存刷新逻辑依然能独立运行。

## 4. 差距分析：当前项目 vs 大厂标准

通过对比，当前项目的 `executeDraw` 实现主要在以下三个维度与大厂标准存在差距：

| 维度 | 当前项目实现 | 大厂标准实现 | 差距风险 |
| :--- | :--- | :--- | :--- |
| **状态流转** | 依赖 JVM 内存（Spring 事件） | 依赖持久化介质（MQ / 本地消息表） | 进程崩溃或 Pod 重启会导致内存中的履约任务丢失。 |
| **补偿方向** | 单向流转（Redis -> DB） | 双向闭环（含反向回撤逻辑） | DB 事务失败后，Redis 库存无法自动找回，造成“库存流失”。 |
| **幂等深度** | 短效防抖（Redis 1s 过期） | 强幂等（DB 唯一键 / 幂等流水表） | 客户端超时重试可能导致绕过 Redis 锁产生重复扣费。 |

---

## 5. 行业最佳实践总结

在大厂看来，没有 100% 的一致性，只有 100% 的**可恢复性**。

**推荐给中小型系统的落地方案优先级：**
1. **DB 唯一索引**：最简单且最有效的强幂等手段。
2. **反向 Lua 补偿**：在 Catch 块中回填 Redis，解决 95% 的即时一致性问题。
3. **本地消息表 + 定时扫表**：解决最后 5% 的异步履约可靠性问题，成本远低于引入 RocketMQ 事务消息。
