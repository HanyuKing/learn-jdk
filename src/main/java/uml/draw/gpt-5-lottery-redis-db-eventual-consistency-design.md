# 抽奖高并发场景下的 Redis 与 DB 最终一致方案

## 1. 目标

前提很明确：

1. 抽奖链路要求高并发。
2. 不希望在 `executeDraw` 里做多次 DB 更新。
3. 仍然希望系统最终可对账、可恢复、不超发、不重复发奖。

在这个前提下，最合适的方案不是“Redis 和 DB 同时保存实时库存，然后定时比较聚合值”，而是：

**Redis 承担实时扣减真相，DB 承担最终账本真相，对账单位从“库存值”切换为“单笔抽奖事件”。**

这套方案追求的是：

1. 热路径只打 Redis，吞吐高。
2. DB 异步落账，写入压力可控。
3. 出现失败时可以按 `drawId/requestId` 精准补偿。
4. 不依赖“当前 Redis 库存 == 当前 DB 库存”这种不成立的判断。

---

## 2. 先说结论

如果要高并发，就不要追求下面这个目标：

> 活动进行中，Redis 的 `stock/issued` 与 DB 的 `stock/issued` 任意时刻严格相等

这个目标在双写体系里不现实。

高并发下真正应该追求的是：

1. **Redis 是线上判定真相**
   - 抽不抽得中
   - 还有没有库存
   - 用户还有没有次数
   - 当前阶段配额够不够
   - 保底是否触发

2. **DB 是最终账本真相**
   - 某次抽奖是否真实发生
   - 最终给了什么奖
   - 这次抽奖是否已持久化
   - 这次抽奖是否需要补偿

3. **对账只对事件，不对聚合库存**
   - 对账 `drawId/requestId`
   - 不对账 `count(db) == issued(redis)`

---

## 3. 为什么不能再用聚合值对账

当前 `LotteryReconcileService#reconcileActivity` 的问题不只是实现粗糙，而是思路本身不对。

### 3.1 聚合值天然不是同一时刻

只要库存还在变化，下面两个值就不可能保证来自同一快照：

1. `dbIssued = count(draw_record)`
2. `redisIssued = get(issued_key)`

哪怕差 5ms，也可能被一笔新抽奖打断。

### 3.2 聚合值不能表达中间状态

高并发抽奖不是“扣库存后立刻全链路完成”，而是会经历：

1. Redis 预占成功
2. DB 待落账
3. DB 已落账但履约未完成
4. 履约失败待重试
5. 明确取消并回补

这几个状态里，单纯看库存聚合数字，无法知道：

1. 哪些是正常 in-flight
2. 哪些是卡单
3. 哪些需要补偿
4. 哪些已经重复落库

### 3.3 聚合重建会覆盖在线流量

如果后台线程拿 DB 旧值去覆盖 Redis，就会把在线刚发生的成功预占抹掉。

这不是修复，是二次破坏。

所以高并发下，**对账必须从“聚合值修正”升级为“事件驱动补偿”。**

---

## 4. 新方案的核心原则

## 4.1 原则一：Redis 只做热路径真相

所有影响抽奖结果的实时状态，都放 Redis：

1. 奖品实时库存
2. 奖品阶段已占用数
3. 用户抽奖剩余次数
4. 用户连续未中次数
5. 请求幂等结果
6. 已生成但待落库的抽奖事件

这样 `executeDraw` 才能高并发。

## 4.2 原则二：DB 不参与实时判定

DB 不再承担线上实时判断：

1. 不再让 DB `award.stock` 参与抽奖决策
2. 不再让 DB `count(draw_record)` 参与是否还能中奖的判定
3. 不再让 DB `balance` 参与热路径余额判断

DB 只负责异步落账。

## 4.3 原则三：Redis 与 DB 的一致性靠事件补偿

不再问：

> “两个库存值为什么不一样？”

而改问：

> “这笔已经在 Redis 成功预占的抽奖事件，为什么没有成功进入 DB 账本？”

这时补偿就能精确到单笔事件。

## 4.4 原则四：任何补偿都必须幂等

无论是：

1. 异步落库重试
2. MQ 重复消费
3. 定时补偿重复扫描
4. 回滚 Redis 预占

都必须围绕 `drawId/requestId` 做幂等。

---

## 5. 建议的整体架构

## 5.1 角色划分

### Redis

负责：

1. 实时库存
2. 实时余额
3. 实时保底计数
4. 请求幂等
5. 抽奖结果缓存
6. 待落库事件队列

### DB

负责：

1. 抽奖账本
2. 余额流水账本
3. 履约账本
4. 审计查询
5. 人工补偿依据

### MQ 或 Redis Stream

负责：

1. 抽奖事件异步传递
2. 落库任务解耦
3. 重试消费

如果当前没有 MQ，可以先用 Redis Stream 过渡。

---

## 6. 推荐状态模型

每次抽奖事件都应该有明确状态，不要只靠库存数字推断。

## 6.1 Redis 侧事件状态

建议设计为：

1. `RESERVED`
   - Redis 预占成功
   - 抽奖结果已经生成
   - 待异步落库

2. `PERSISTING`
   - 消费者正在落库

3. `PERSISTED`
   - DB 账本已写成功

4. `CANCELLED`
   - 因 DB 长期落库失败或明确补偿，Redis 预占已回滚

### 注意

这里的状态可以放在 Redis 哈希中，不一定要持久化到 DB。

DB 最终只关心：

1. 这笔事件是否存在
2. 这笔事件的业务结果是什么

## 6.2 DB 侧抽奖记录状态

DB `draw_record` 建议表达为：

1. `PROCESSING`
   - 抽奖账本已落库，待履约

2. `SUCCESS`
   - 履约完成

3. `RETRYING`
   - 履约失败待重试

4. `CANCELLED`
   - 明确取消

不要让 DB `FAIL` 同时承担：

1. 临时失败待重试
2. 永久取消

这两个语义必须拆开。

---

## 7. Redis Key 设计建议

下面只列核心 key。

## 7.1 实时库存与用户状态

1. `jjewelry:lottery:stock:{actId}:{awardId}`
   - 奖品实时剩余库存

2. `jjewelry:lottery:quota:{actId}:{awardId}:{stage}`
   - 当前阶段已占用数

3. `jjewelry:lottery:balance:{actId}:{userId}`
   - 用户剩余抽奖次数

4. `jjewelry:lottery:miss:{actId}:{userId}`
   - 用户连续未中次数

## 7.2 幂等与结果

1. `jjewelry:lottery:request:{requestId}`
   - 请求幂等键
   - 值建议直接存：
     - `PROCESSING`
     - 或 `DONE:{drawId}`

2. `jjewelry:lottery:draw:{drawId}`
   - 单笔抽奖结果详情
   - 包含：
     - `drawId`
     - `requestId`
     - `actId`
     - `userId`
     - `awardId`
     - `awardType`
     - `stage`
     - `status`
     - `createTime`

## 7.3 待落库队列

两种实现方式：

1. MQ
2. Redis Stream / ZSet

如果用 Redis Stream：

1. `jjewelry:lottery:stream:draw`
   - 事件主流

如果用 ZSet 辅助超时扫描：

1. `jjewelry:lottery:pending`
   - score = 预占时间戳
   - member = `drawId`

---

## 8. 热路径设计

高并发的关键是：**一次 Lua 原子完成所有实时变更。**

## 8.1 `executeDraw` 热路径步骤

推荐改成：

1. 校验活动和规则缓存
2. 调 Redis Lua
3. Lua 内原子完成：
   - 幂等校验
   - 余额校验
   - 当前奖项库存校验
   - 阶段配额校验
   - 保底判定相关状态更新
   - 扣库存/扣余额
   - 生成 `drawId`
   - 存 `draw:{drawId}`
   - 写 `request:{requestId}=DONE:{drawId}`
   - 写入 `pending` / `stream`
4. 接口直接返回 Redis 中的抽奖结果

整个热路径不做同步 DB 事务。

## 8.2 为什么必须一次 Lua 完成

如果拆成多个 Redis 命令：

1. 先扣库存
2. 再扣余额
3. 再写结果
4. 再写 pending

任何一步失败都会让热状态碎裂。

只有 Lua 能保证：

1. 要么这一笔完整预占成功
2. 要么这一笔完全没发生

这就是 Redis 侧强原子性。

---

## 9. 异步落库设计

## 9.1 落库内容

消费者从 `stream/MQ/pending` 取到 `drawId` 后，去 Redis 读取 `draw:{drawId}` 详情，然后在 DB 事务中落账：

1. 插入 `draw_record`
2. 插入余额流水
3. 可选插入库存占用流水

### 关键点

这里的 DB 事务只做账本写入，不做实时业务判断。

## 9.2 DB 唯一键要求

至少要保证：

1. `draw_record.draw_id unique`
2. `request_id unique`，如果 `draw_record` 上有这个字段
3. `balance_log (biz_id, biz_type) unique`

这样即使：

1. 消费重复
2. 网络超时重试
3. 同一事件重复投递

也不会写出第二笔真实账。

## 9.3 成功后的 Redis 状态更新

DB 提交成功后：

1. 把 `draw:{drawId}.status` 更新为 `PERSISTED`
2. 从 `pending` 中移除
3. 或标记消息已消费

这一步可以是最终一致，不影响主账正确性。

---

## 10. 对账与补偿方案

这是整套方案的核心。

## 10.1 对账单位必须是 `drawId`

定时任务不再做：

1. 比较 `dbIssued`
2. 比较 `redisIssued`
3. 不等就重建库存

而要做：

1. 扫描 Redis `pending`
2. 逐笔检查这笔 `drawId` 是否已写入 DB

## 10.2 对账逻辑

### 情况一：DB 已存在该 `drawId`

说明只是 Redis 清理不及时。

处理：

1. 标记 `PERSISTED`
2. 从 `pending` 移除

### 情况二：DB 不存在该 `drawId`，但未超时

说明大概率只是异步链路稍慢。

处理：

1. 不补偿
2. 继续等待

### 情况三：DB 不存在该 `drawId`，且已超时

说明这笔 Redis 预占已经失联，需要补偿。

处理：

1. 尝试重新投递落库
2. 达到阈值仍失败，再执行反向 Lua 补偿：
   - `stock + 1`
   - `balance + 1`
   - `quota - 1`
   - `missStreak` 回滚或改写
   - `draw:{drawId}.status = CANCELLED`
   - `request:{requestId}` 保持幂等可识别状态

## 10.3 补偿一定要基于单笔事件

只有这样才能知道：

1. 应该回滚哪个奖品
2. 应该回滚哪个用户的余额
3. 应该回滚哪个阶段配额
4. 这笔事件是否已经被补偿过

如果仍然按“聚合库存差值”补，会出现：

1. 回滚错用户
2. 回滚错奖品
3. 重复回滚
4. 无法处理部分成功

---

## 11. 失败场景逐一分析

## 11.1 Lua 执行前应用崩溃

结果：

1. 没有预占成功
2. 没有 `drawId`
3. 没有 `pending`

无需补偿。

## 11.2 Lua 成功后，应用在返回前崩溃

结果：

1. Redis 已预占成功
2. `drawId` 已存在
3. `pending` 已存在
4. 客户端可能没收到响应

处理：

1. 依赖 `requestId` 幂等
2. 重试请求时直接返回已有 `drawId` 结果
3. 异步流程继续落库

## 11.3 Lua 成功，消息投递失败

结果：

1. Redis 已预占
2. DB 还没有记录

处理：

1. `pending` 扫描任务重试投递
2. 超时仍失败再补偿

## 11.4 消费者收到消息，DB 写一半失败

处理：

1. DB 事务整体回滚
2. 事件仍保持 `RESERVED`
3. 后续继续重试

只要唯一键设计正确，重复消费不会破坏账本。

## 11.5 DB 永久不可写

处理：

1. 暂时积压 pending
2. 超过业务容忍阈值后，统一补偿 Redis 预占

### 注意

这里要么选择“继续保留中奖结果等待恢复”，要么选择“超时自动取消并回补”。

这个策略必须业务先定。

## 11.6 履约失败

履约失败不等于回滚库存。

推荐处理：

1. `draw_record` 标记 `RETRYING`
2. 继续重试履约
3. 只有人工或业务明确取消时，才单独做取消补偿

---

## 12. 关键设计细节

## 12.1 `requestId` 必须是真幂等，不是 1 秒防抖

当前只做 `SET NX EX 1s` 不够。

建议：

1. `request:{requestId}` 先写 `PROCESSING`
2. Lua 成功后改成 `DONE:{drawId}`
3. 后续重复请求：
   - 看到 `DONE:{drawId}` 直接返回原结果

必要时这个关系也可以异步落 DB。

## 12.2 `drawId` 要在 Redis 热路径内生成

因为补偿和落库都要围绕这笔事件做。

如果 `drawId` 是 DB 插入后才有，就没法在 Redis 侧精准补偿。

## 12.3 `missStreak` 回滚要谨慎

这个字段如果在 Lua 里被更新了，补偿时也要考虑。

推荐做法：

1. 在 `draw:{drawId}` 中记录补偿前后必要状态
2. 反向 Lua 根据事件内容精确回滚

否则会出现：

1. 库存回滚了
2. 用户保底计数没回滚

从而影响下一次抽奖结果。

## 12.4 阶段配额也必须纳入事件补偿

如果阶段配额是实时扣减的，那么一笔落库失败的事件，除了库存和余额，也要回滚配额占用。

否则会出现：

1. 库存恢复了
2. 配额没恢复

导致用户看起来还有库存，但永远抽不中。

---

## 13. DB 表设计建议

如果沿用当前表，也建议补充或调整几项。

## 13.1 `draw_record`

建议至少增加：

1. `request_id`
2. `persist_status` 或更清晰的业务状态
3. `award_snapshot`
4. `stage_snapshot`

其中：

1. `draw_id unique`
2. `request_id unique`

## 13.2 `balance_log`

继续保留唯一键：

1. `(biz_id, biz_type)` unique

这里 `biz_id` 可以直接用 `drawId`。

## 13.3 可选：增加库存占用流水表

如果后面排障要求高，建议新增：

1. `lottery_stock_reservation_log`

记录：

1. `draw_id`
2. `activity_id`
3. `award_id`
4. `delta`
5. `reservation_status`

这样以后不必再从聚合值倒推问题。

---

## 14. `reconcileActivity` 在新方案里应该怎么改

它应该从“库存重建器”变成“事件补偿扫描器”。

建议职责如下：

1. 扫描 Redis `pending`
2. 按 `drawId` 检查 DB 是否已落账
3. 未落账则重试投递
4. 超时则发起单笔补偿
5. 记录异常事件供人工查看

### 不再做的事情

1. 不再比较 `dbIssued` 和 `redisIssued`
2. 不再根据聚合值判断偏差
3. 不再把 DB 聚合库存覆盖回 Redis

---

## 15. 这套方案的优点

## 15.1 优点

1. 热路径完全 Redis 化，吞吐高。
2. DB 压力从同步写变成异步账本写，峰值更稳。
3. 对账从模糊聚合修正变成精准事件补偿。
4. 出问题时可定位到单笔 `drawId`，排障成本低。
5. 不需要要求 DB 与 Redis 任意时刻聚合值相等。

## 15.2 代价

1. 系统复杂度提升。
2. Redis 持久化要求更高。
3. 必须补齐幂等、补偿、超时扫描机制。
4. 需要接受 DB 不是实时查询真相，只是最终账本。

---

## 16. 必须满足的基础设施要求

如果要走这条路，有几个前提必须满足。

## 16.1 Redis 必须可恢复

至少要求：

1. AOF
2. 合理刷盘策略
3. 主从或哨兵/集群

否则 Redis 一丢，线上真相就没了。

## 16.2 补偿任务必须稳定

至少要有：

1. pending 扫描任务
2. 重试投递任务
3. 超时取消补偿任务

这些任务的可靠性直接决定最终一致性。

## 16.3 幂等键不能缺

没有 `requestId/drawId` 唯一锚点，任何最终一致方案都会退化成人工对账。

---

## 17. 给当前代码的落地建议

如果基于当前项目演进，我建议分三步走。

## 第一步：先把聚合对账思路停掉

也就是：

1. 不再使用“比较 `dbIssued` 与 `redisIssued`”作为修正依据
2. 不再做按聚合值 `rebuildFromDB`

## 第二步：补齐 Redis 事件模型

新增或改造 Redis 侧能力：

1. `requestId -> drawId` 结果幂等
2. `draw:{drawId}` 详情
3. `pending` 待落库集合
4. 单笔事件补偿 Lua

## 第三步：把 DB 改成异步账本

改造点：

1. `executeDraw` 不再同步做完整 DB 事务
2. 异步消费者写 `draw_record + balance_log`
3. 履约链路独立于库存热状态

---

## 18. 最终建议

如果你的优先级是：

1. 高并发优先
2. Redis 可承载实时状态
3. 可以接受最终一致

那就不要再用“库存聚合值对账”这条路。

正确方向是：

> Redis 负责实时预占，DB 负责最终账本，`reconcile` 负责单笔事件补偿。

这套方案的关键不是“把 Redis 和 DB 的数字调成一样”，而是：

> 保证每一笔已经在 Redis 成功预占的抽奖，要么最终成功进入 DB，要么被精确、幂等地回滚。

只要这条做到，系统就是可控的最终一致。

1. Lua 输入输出协议
2. Redis key/field 结构
3. pending 扫描状态机
4. 具体失败时序图

这样后面改代码会更顺。
