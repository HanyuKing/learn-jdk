# gemini-lottery-executeDraw-analysis.md

## 1. 核心流程概述

`LotteryDrawServiceImpl#executeDraw` 采用的是典型的“缓存预扣 + 异步落盘 + 最终一致”架构。

**核心步骤：**
1. **防重与加锁**：通过 `requestId` 幂等校验和 `userId` 分布式锁，确保高并发下同一用户请求的串行化。
2. **规则与状态加载**：从本地缓存加载规则，从 Redis 批量读取奖品库存、已发放数等实时状态。
3. **决策引擎**：在内存中完成中奖计算（含保底逻辑）。
4. **Redis 原子扣减**：利用 Lua 脚本原子性地扣减库存、增加已发放数。
5. **DB 事务落盘**：异步或同步扣除用户余额并记录抽奖流水（状态为 `PROCESSING`）。
6. **异步履约**：通过 `ApplicationEvent` 异步触发奖品发放及 DB 库存扣减。

## 2. DB 与 Redis 数据一致性分析

### 2.1 存在的一致性风险

1. **Redis 成功后 DB 失败（无回滚补偿）**：
   - **场景**：Redis Lua 脚本执行成功（库存已扣），但后续 `transactionService.drawAndDeduct` 因数据库异常、余额不足或其他原因回滚。
   - **后果**：Redis 中的库存被永久扣除，但用户未获得抽奖记录，导致“库存流失”。
   - **当前状态**：代码中未见对 Redis 扣减的反向补偿逻辑。

2. **分布式事务缺失**：
   - Redis 和 MySQL 属于不同的资源，当前的实现依赖于代码顺序执行，并非强一致性分布式事务。
   - 在高并发或极端宕机情况下，可能出现 Redis 状态更新而 DB 记录未落盘的情况。

3. **异步履约延迟导致的“逻辑超卖”**：
   - 真正的 DB 库存（`jewelry_lottery_award.stock`）是在异步履约阶段扣减的。
   - 虽然 Redis 挡住了物理超卖，但如果异步消息堆积或消费失败，会导致 DB 账面库存与 Redis 实时库存长期不一致，给财务对账带来困扰。

4. **幂等性不足**：
   - `requestId` 仅在 Redis 中保留 1 秒（通常实现），且成功后未持久化结果。
   - 若客户端超时重试，可能在 1 秒后进入第二次抽奖流程，导致重复扣费。

### 2.2 解决方案建议

- **增加反向补偿**：在 `executeDraw` 的 `catch` 块中，识别出 Redis 已扣减但事务未成功的场景，调用 Lua 脚本回滚 Redis 库存。
- **本地消息表/事务消息**：将“Redis 占用”作为一个事件记入 DB 事务。如果事务成功但后续流程断掉，通过后台任务根据消息表进行 Redis 状态校准。
- **强化幂等**：将 `requestId` 与抽奖结果绑定，并持久化到数据库中（如唯一索引表），确保同一 `requestId` 永远只返回相同结果。

## 3. 高并发性能问题分析

### 3.1 性能设计亮点

1. **Redis Lua 原子化**：将库存校验、配额校验、库存扣减、已发放数累加、保底计数清零等多个操作封装在 Lua 脚本中，减少了多次 RTT（往返时延），保证了操作的原子性。
2. **用户级分布式锁**：锁粒度细化到 `userId`，避免了全局锁竞争，极大提升了并发处理能力。
3. **前置校验（Fast-fail）**：在进入复杂逻辑前先校验活动时间、幂等、用户余额，过滤无效请求。
4. **异步履约**：将耗时的发券、发货、通知等操作移出主事务，缩短了数据库事务持有时间。

### 3.2 潜在瓶颈

1. **Redis 热点 Key**：
   - 虽有用户锁，但奖品库存、活动规则等数据仍存储在特定的 Redis Key 中。在超大规模促销（如秒杀）下，单分片 Redis 可能会面临压力。
2. **DB 事务压力**：
   - 尽管是异步履约，但每笔抽奖仍需写入 `draw_record` 和扣减 `lottery_remaining`。高并发下，数据库连接池和磁盘 IO 可能成为瓶颈。
3. **规则计算开销**：
   - `calcAllowedMap` 等逻辑在每次抽奖时都重新计算。如果奖品数量极多，虽然在内存中，也会消耗一定 CPU。

### 3.3 优化建议

- **Redis 分段库存**：对于爆款奖品，可以将库存拆分到不同的 Redis Key 中存储，分散压力。
- **批量落盘**：对于极高并发场景，可以考虑将 `draw_record` 先写入高性能日志或消息队列，再由消费者批量写入 DB（需权衡数据丢失风险）。
- **缓存计算结果**：`calcAllowedMap` 的结果在同一阶段（Stage）内是稳定的，可以考虑基于 Stage 进行短时间缓存。

## 4. 总结与改进方案

| 维度 | 问题点 | 推荐解决方案 |
| :--- | :--- | :--- |
| **一致性** | Redis 扣减后 DB 事务回滚 | 引入 catch 块中的反向补偿 Lua 脚本，或使用本地消息表确保闭环。 |
| **一致性** | 异步履约失败导致 DB 库存不准 | 建立完善的重试机制，并在对账系统中以 `draw_record` 为准修复 Redis 状态。 |
| **一致性** | 幂等性弱 | 建立 `requestId` 持久化表，或将 `requestId` 作为 `draw_record` 的唯一索引。 |
| **性能** | DB 写入瓶颈 | 采用分库分表，或对非核心抽奖采用“先入队、后落盘”的缓冲策略。 |
| **性能** | Redis 单点压力 | 实施库存分片存储，并对活动规则等静态数据使用强一致性本地缓存。 |

通过上述优化，可以进一步提升抽奖系统的健壮性和承载力，确保在极端高并发下既“跑得快”又“算得准”。
