# 0.01 元国际化改造方案（统一最低价）

> **决策（2026-06-23 用户确认）**
> 1. Nacos 解析失败 / 配置缺失 → fallback **0.5 USD**。
> 2. 全仓所有 `new BigDecimal("0.01")` 业务硬编码统一替换为**同一个可配置最低价**，作者一口价、迷你支付阈值、折扣下限、告警阈值等都走这一个值。
> 3. order-java 创建 / 改价商品时强制校验 ≥ 最低价。
> 4. **硬下限 0.5**：`NacosOrderCommonConfig.getMinAmount()` 内置 floor，即便 Nacos 上有人误配置 0.01，运行时也强制抬到 0.5；任何业务点（包括 `authorFreeSampleQuotaConfig.freeDiscount.discountValue` 等其他既有配置）取到的最终金额都不允许低于 0.5。

---

## Context

仓库 `zaohaowu_i18n` 是国内造好物的 USD 国际化版本（Stripe `currency: usd`）。代码里大量 `new BigDecimal("0.01")` 原意是「人民币 1 分」，被复用为四类语义：

- 作者免费打样 / Want It 一口价
- 折扣后下限（防止折后 ≤ 0）
- 「迷你支付」识别（`isMiniPaid` → 跳过退款 / 想要发券）
- 风控告警阈值（`< 0.01` 触发告警）

国际化后这些值不再适合写死。本轮**统一**用一个 Nacos 可配置的最低价（默认 0.5 USD）覆盖所有业务点。

## 改造方案

### 1. 新建独立 Nacos data-id `order-common.json`

**为什么不复用 `order-config.yml`**：现有 `order-config.yml` 里已经堆了很多关键配置（白名单、商品默认数据、作者免费打样配额、采购折扣率等），任何一项漏写都会全量回退默认值，本轮新增字段挤进去风险高。新开一个 data-id 隔离，影响面只限本次新增。

**新 data-id**：
- group: `order`
- dataId: `order-common.json`
- type: JSON
- autoRefreshed: true

**新建配置类**（不动现有 `NacosConfig.java`）：

`order-java/trade-order-service/src/main/java/cn/mathmagic/jjewelry/business/common/config/NacosOrderCommonConfig.java`

```java
@Slf4j
@Component
@RefreshScope
@Getter
@Setter
@NacosConfigurationProperties(
        groupId = "order",
        dataId = "order-common.json",
        type = ConfigType.JSON,
        autoRefreshed = true
)
public class NacosOrderCommonConfig {

    private static final BigDecimal HARD_FLOOR = new BigDecimal("0.5");

    /**
     * Nacos JSON 示例：
     * {
     *   "minAmount": {
     *     "USD": 0.5,
     *     "CNY": 0.5
     *   }
     * }
     */
    private Map<String, BigDecimal> minAmount;

    @Value("${app.activeCurrency:USD}")
    private String activeCurrency;

    /** 当前激活币种最低金额。Nacos 不可用 / 缺字段 / 配置值低于硬下限时返回 0.5。 */
    public BigDecimal getMinAmount() {
        Map<String, BigDecimal> map = this.minAmount != null ? this.minAmount : Collections.emptyMap();
        BigDecimal v = map.get(activeCurrency);
        // 硬下限保护：即便 Nacos 配错（如填了 0.01），运行时也强制不低于 0.5
        if (v == null || v.compareTo(HARD_FLOOR) < 0) {
            return HARD_FLOOR;
        }
        return v;
    }
}
```

**Nacos 控制台中 `order-common.json` 的初始内容**（首次发布时填，标准 JSON）：

```json
{
  "minAmount": {
    "USD": 0.5,
    "CNY": 0.5
  }
}
```

> 注意：当前实现用 `@NacosConfigurationProperties` 直接绑定 `minAmount` 字段，不再通过 `@NacosValue String` 手动解析。Nacos 控制台必须按上面的标准 JSON 发布。

**跨服务复用**：
- `jjewelry-java`、`reward-task-center` 各自新建一个同样的 `NacosOrderCommonConfig`（拷贝粘贴），挂同一 group + dataId，配置天然同源。
- 不复用 `trade-order-common`，避免跨服务依赖。
- 业务点直接 `@Resource NacosOrderCommonConfig minAmountConfig; minAmountConfig.getMinAmount();`，热路径无锁读 `volatile`，不重复解析 JSON。

### 2. `OrderUtil.isMiniPaid` 保留静态调用面

`jjewelry-api-service/.../trade/util/OrderUtil.java` 当前是纯 static 工具：

- 为了最小化代码改动，不新建 `OrderAmountChecker`，不改调用方，保留 `OrderUtil.isMiniPaid(...)` 静态方法签名。
- `OrderUtil` 内部接入 `NacosOrderCommonConfig`：
  - 优先方案：`OrderUtil` 增加 `@Component`，通过非静态 setter 注入 `NacosOrderCommonConfig` 到 `private static volatile` 字段，静态 `isMiniPaid` 方法里读取 `minAmountConfig.getMinAmount()`。
  - 若注入尚未完成 / Spring 上下文不可用，方法内 fallback `new BigDecimal("0.5")`，与 `NacosOrderCommonConfig` 的硬下限保持一致。
- 调用方无需变更，只做 grep 验证仍然只有既有静态调用：
  - `PurchaseOrderPayAndRefundConsumer.java:186`
  - `WantItActivityService.java:1593`
- 语义保持：`payAmount`（不含运费 / 改价情况下含运费）等于 `minAmountConfig.getMinAmount()` 判定为 mini-paid。

### 3. 商品创建 / 改价价校验

`PublishProductProxy.java`：
- `validateBasicParams` 由 `static` 改实例，类自身注入 `NacosOrderCommonConfig`。
- 行 143 `price == null` 校验之后追加：
  ```java
  BigDecimal min = minAmountConfig.getMinAmount();
  if (productDto.getPrice().compareTo(min) < 0) {
      throw new ServiceException(ErrorCodeEnum.PARAM_ERROR,
          OrderI18nKeys.PUBLISH_PRODUCT_PROXY_PRICE_BELOW_MIN, min);
  }
  ```
- `createCmd.getSkuDtoList()` 校验段（行 165 之后）追加遍历，对每个 `SkuDto.price < min` 抛同一个 i18n key（参数化最小值）。
- `validateUpdateParams` / `validateBasicUpdateParams`（如有）同步加。
- `SkuInnerController.updatePrice`（行 90-95）单独的改价入口同样加 `>= min` 校验。

控制器 (`ProductInnerController.addProduct/batchAddProduct/updateProduct`、`ProductController.addProduct`) 都走 `PublishProductProxy.validate*`，无需重复加。

### 4. i18n key

新增（zh + en 双语，落 `messages_zh.properties` / `messages_en.properties`，并在 `OrderI18nKeys` 注册）：

| Key | zh | en |
|---|---|---|
| `PUBLISH_PRODUCT_PROXY_PRICE_BELOW_MIN` | `商品价格不能低于最低金额 {0}` | `Product price must be at least {0}` |

同时改既有告警文案中带「0.01」字样的：
- `MarketingExtService` 飞书告警「product 价或活动价小于 0.01」 → 拼最小值动态化（这是飞书内部告警，不强制走 i18n，但要去掉硬编码字样）。
- `PriceService` 告警「支付价小于 0.01」同上。

### 5. 风险 & 影响面

- **`isMiniPaid` 阈值变化**：旧线上的 0.01 USD 测试单不再被识别为 mini-paid，会进入正常退款 / 想要弹窗流程。如果 DB 里仍有这些单，需要数据修复或灰度。
- **作者活动一口价跳到 0.5 USD**：所有命中作者促销的商品成交价会从 0.01 变 0.5。要先和业务确认能接受。
- **既有 `authorFreeSampleQuotaConfig` Nacos 配置**：线上配置中 `freeDiscount.discountValue` 历史值是 0.01。本次改造做两层防护：
  - 代码侧：`buildDefaultAuthorFreeSampleQuotaConfig` 兜底改用 `minAmountConfig.getMinAmount()`；`parseAuthorFreeSampleQuotaConfig` 解析后对 `discountValue < minAmount` 强制抬到 minAmount。
  - 配置侧：上线时把 Nacos 中 `discountValue: 0.01` 调到 0.5（或删字段让代码兜底）。
  - 即使配置侧忘改，代码侧硬下限保护也能确保实际下发值 ≥ 0.5。
- **告警阈值同步抬升**：`< 0.5` 才告警，原本 0.01-0.49 之间的异常不再上报。如果业务希望保留更敏感的告警，可以未来把 `MinAmountConfig` 拆「业务最小价」与「告警阈值」两字段；本轮按用户「统一一个值」的指示先合并。
- **DiscountCalculator / PriceService 在算价热路径**：`getMinAmount()` 走 `volatile` 缓存读 + JSON 串相等比对，常态零开销，仅 Nacos 推送变更时重新解析一次。

## 验证方案

1. **单测**
   - `NacosOrderCommonConfig.getMinAmount()` fallback：Nacos 字符串空 / 非法 JSON / 缺 USD → 返回 0.5。
   - `OrderUtil.isMiniPaid` 保留静态调用面，断言阈值随 `NacosOrderCommonConfig` 变化；配置不可用时 fallback 0.5。
   - `PublishProductProxy.validateBasicParams`：`price=0.4` 抛 PARAM_ERROR；`price=0.5` 通过；SKU 任一价 < 0.5 同样抛错。
   - `DiscountCalculator`：折后 < 0.5 应被 `.max` 抬到 0.5。

2. **本地联调**
   - `trade-order-service` 启动，Nacos 下发 `minAmount.USD=1.0`，调商品创建 → `price=0.8` 拒、`price=1.0` 过；调商详 → 作者优惠商品 `discountPrice=1.0`；告警阈值随之抬升。
   - 制造一笔 1.0 USD 实付订单 → `isMiniPaid` true；1.5 USD → false。

3. **回归**
   - `order-java` / `jjewelry-java` 全量单测。
   - `cicd/scripts/check_i18n_keys.sh` 校验 zh/en key 同步。
   - 端到端：作者活动下单 → 退款 / 想要发券链路是否按预期跳过。
